# Testing

## Rust tests

### Where they live

Tests are spread across the three crates:

| Location | What it covers |
|---|---|
| `rust/core/src/discrete_log.rs` | DiscreteLogSolver unit tests |
| `rust/core/src/range_proof.rs` | Proof generation and verification round-trip tests |
| `rust/core/tests/cross_version_compat.rs` | Bulletproofs cross-version compatibility (see below) |
| `rust/mobile/src/shared.rs` | Input validation and flat-buffer utilities |

### How to run them

Run all Rust tests across the whole workspace:

```bash
cargo test --manifest-path rust/Cargo.toml --workspace
```

To run only one crate's tests:

```bash
cargo test --manifest-path rust/core/Cargo.toml
cargo test --manifest-path rust/mobile/Cargo.toml
```

To run a specific test by name:

```bash
cargo test --manifest-path rust/Cargo.toml --workspace <test_name>
```

## Cross-version compatibility tests

**File:** `rust/core/tests/cross_version_compat.rs`

These tests verify that range proofs generated by the current library can be verified by the on-chain verifier, which uses `bulletproofs` v4.0.0, even though the library's Rust dependency uses v5.

What they check:

- Proofs are produced with the DST `b"AptosConfidentialAsset/BulletproofRangeProof"` and the generator parameters that match the on-chain verifier's expectations.
- The serialised proof bytes are stable across library versions.

Why they matter: the on-chain verifier cannot be updated without a network upgrade. If proof format or parameters drift, transactions will fail on-chain but pass local tests. The cross-version tests catch this class of regression early.

**These tests must pass before any release.** Changes to `rust/core/src/range_proof.rs` that affect the DST, generator parameters, or serialisation format require a coordinated upgrade with the Aptos network team.

## Mobile shared module tests

**File:** `rust/mobile/src/shared.rs`

`shared.rs` contains unit tests for:

- **Input validation** — checks that out-of-range values, empty inputs, and mismatched counts are rejected with the expected error codes before any crypto work is done.
- **Flat-buffer splitting** — verifies that arrays of variable-length byte slices encoded as a flat buffer with an explicit count are correctly split back into individual items.

These tests run as part of the full workspace test run and do not require a device or emulator.

## JavaScript tests

### How to run

```bash
npm test
```

This invokes `expo-module-scripts` test runner.

### What to test

The JS test suite should cover:

- Public API contracts: correct output types for valid inputs.
- Input validation: error paths for invalid arguments (wrong types, out-of-range values, mismatched array lengths).
- WASM initialisation: the WASM module loads and the exported functions are callable in a Node.js environment.
- Native module interface: that `index.native.ts` validates inputs and passes correctly shaped arguments to the JSI module.

Because the core cryptographic correctness is covered by Rust tests, JS tests focus on the TypeScript layer: types, validation, and module loading.

## CI test matrix

The `.github/workflows/ci.yml` workflow runs on every push and pull request to `main`:

| Job | What it runs |
|---|---|
| `lint-and-typecheck` | `npm run lint` + `npm run typecheck` (Node only, no Rust needed) |
| `test-rust` | `cargo test --workspace` with Swatinem/rust-cache |
| `test-js` | `npm test` via expo-module-scripts |
| `build` | Full `npm run build` with Rust cache |

All four jobs must pass for a PR to be mergeable.

## Adding a new test

### Rust unit test

Add a `#[cfg(test)]` module to the source file where the function under test lives:

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn my_new_test() {
        // ...
    }
}
```

Run it in isolation with:

```bash
cargo test --manifest-path rust/Cargo.toml --workspace my_new_test
```

### Rust integration test

Add a new file under `rust/core/tests/`. Integration tests have access to the crate's public API only (the same surface as external callers), which makes them appropriate for end-to-end crypto tests.

If the test verifies on-chain compatibility (proof format, DST, serialisation), add it to `cross_version_compat.rs` or create a new file in the same directory with a descriptive name.

### JS test

Add a test file or case within the existing test setup under `examples/expo` or alongside the relevant source file in `src/`. Follow the patterns already present in the test suite. Prefer testing the behaviour visible to callers of the public API rather than internal implementation details.
